## What It Does

Driver Pulse ingests raw sensor and financial data from a driver's shift and produces two things:

1. **Safety signal detection** — identifies moments during trips where motion and audio signals overlap to suggest stress, conflict, or aggressive traffic conditions.
2. **Earnings velocity tracking** — transforms cumulative earnings from a static end-of-day number into a live pace indicator with goal projection.

- Python 3.8+
- pandas, numpy (`pip install pandas numpy`)

## Pipeline Architecture

```
Raw Inputs
    accelerometer_data.csv  ──┐
    audio_intensity_data.csv ─┤
    trips.csv                 ├──▶  STEP 1: Feature Extraction
    driver_goals.csv          │         (g-force magnitude, audio classification)
    earnings_velocity_log.csv ┘
                                         │
                                         ▼
                                  STEP 2: Event Detection
                                    Motion:  harsh_braking | moderate_brake
                                    Audio:   audio_spike
                                         │
                                         ▼
                                  STEP 3: Signal Fusion
                                    Overlap window: 120s
                                    conflict_moment = motion ∩ audio
                                    combined_score = 0.55×motion + 0.45×audio
                                         │
                                         ▼
                                  STEP 4: Earnings Velocity
                                    velocity = cumulative_earnings / elapsed_hours
                                    projected = earned + (velocity × remaining_hours)
                                         │
                                         ▼
                              Output Logs + Dashboard
```

## Detection Thresholds

| Signal | Threshold | Rationale |
|--------|-----------|-----------|
| Harsh braking | horizontal g-force ≥ 2.5 m/s² | Exceeds typical urban deceleration (~1.5 m/s²); consistent with ISA/NHTSA harsh-event definitions |
| Moderate brake | horizontal g-force ≥ 1.8 m/s² | Elevated but not extreme; flags for driver awareness without alarm |
| Audio spike | classification = "argument" OR dB ≥ 70 + sustained ≥ 20s | Avoids false positives from short loud sounds (door slams, horns) |
| Conflict moment | combined_score ≥ 0.75 | Requires both signals to be strong; reduces single-signal false positives |
| Signal fusion window | 120 seconds | Conservative window — motion and audio events in the same ~2min window suggest the same incident |

### Sensitivity vs False Positive Trade-offs

The system is tuned toward **lower false positives** over maximum sensitivity because:
- Drivers should not feel surveilled or mistrusted
- A false positive (flagging a non-event) erodes trust faster than a missed event
- The `combined_score` filter ensures `conflict_moment` only fires when *both* signals are elevated

To increase sensitivity (catch more events), lower `HARSH_BRAKE_THRESHOLD` to 1.5 and `AUDIO_HIGH_DB` to 65.

## Known Limitations

| Limitation | Impact | Mitigation |
|-----------|--------|------------|
| Sensor data covers only TRIP001–TRIP030 (30/220 trips) | 190 trips have 0 flags — not because nothing happened, but no sensor readings | Dashboard shows coverage notice; pipeline marks trips as "no sensor data" |
| Audio classification is pre-labelled in dataset | In production, this would require on-device audio energy bucketing (no transcript) | Privacy-safe: only intensity dB + duration used, never content |
| Earnings velocity assumes constant rate | Mid-shift pace changes (surge hours, dead zones) not modelled | A rolling 30-min velocity window would be more accurate in production |
| No GPS-based context | Cannot distinguish hard braking from a speed bump vs emergency stop | Could be improved with road type overlay from Uber's mapping stack |

## Privacy Design

Driver Pulse is privacy-safe by construction:

- **No audio content** is recorded or transmitted. Only aggregated dB intensity values are used.
- **No location tracking** beyond what already exists in trip records.
- **No behavioural scoring** of drivers — flags are informational, not punitive.
- All signal processing runs **on-device** (edge) for real-time detection; cloud sync is for post-shift summaries only.

## Output File Formats

### flagged_moments.csv

| Column | Type | Description |
|--------|------|-------------|
| flag_id | string | Unique event ID (FLAG001…) |
| trip_id | string | Trip the event occurred in |
| driver_id | string | Driver |
| timestamp | datetime | Absolute time of event |
| elapsed_seconds | int | Seconds since trip start |
| flag_type | enum | harsh_braking / moderate_brake / audio_spike / conflict_moment / sustained_stress |
| severity | enum | low / medium / high |
| motion_score | float [0–1] | Normalised motion signal strength |
| audio_score | float [0–1] | Normalised audio signal strength |
| combined_score | float [0–1] | Weighted fusion score |
| explanation | string | Plain-English reason shown to driver |
| context | string | Raw signal labels for engineering audit |

| Column | Type | Description |
